# Role
あなたは経験豊富なフルスタックエンジニアです。
私のポートフォリオ用に、シンプルかつ堅牢な「社内備品管理システム」をゼロから構築してください。

# プロジェクト名: AssetFlow

# 技術スタック
- **Front**: React 18 + TypeScript + Tailwind CSS + Vite
- **Back**: Python FastAPI + SQLAlchemy
- **DB**: PostgreSQL (Docker Compose使用)
- **Infrastructure**: DevContainer対応 (.devcontainer/devcontainer.json)

# 機能要件（2画面集中）

## 1. 備品一覧画面
- **表示**: テーブル形式（備品名、カテゴリ、現在の在庫数、状態）。
- **検索**: 備品名による部分一致検索。
- **在庫警告**: 在庫5個以下で在庫数を赤文字表示。
- **UIアクション（こだわり）**:
    - 通常時はボタンを非表示にし、行を**マウスホバーした時のみ**、右端に「この備品を借りる」ボタンを浮き上がらせること（Tailwindの `group-hover` 等を使用）。
    - 在庫0の備品は赤字で強調し「申請不可」と表示。ホバーしてもボタンは表示させないこと。
    - ボタン押下で「申請画面」へ遷移。

## 2. 申請（借りる）画面
- **表示**: 選択した備品の情報をヘッダーに表示（一覧画面からの遷移時に情報を引き継ぐか、IDで再取得）。
- **フォーム**: 申請者名、使用期間（開始・終了）、使用理由、申請数量を入力。
- **ロジック**: 申請ボタン押下時、在庫チェックを行い、不足していればトースト等で警告。
- **仕様**: 申請時点では在庫を減らさず、申請データのみを作成する（保留状態）。

# ビジネスロジック・実務要件
- **在庫1以上ルール**: 備品登録時（初期データ含め）は必ず在庫1以上とする。在庫0での新規登録は不可。
- **排他制御**: 在庫確認と申請データ作成において、トランザクション管理と排他制御（悲観ロック等）を考慮した堅牢な実装にすること。
- **共通レスポンス形式**: すべてのAPIは `{ "success": true, "data": ..., "error": null }` の形式でレスポンスを返すこと。
- **エラーハンドリング**: 入力バリデーション、DBエラー、在庫不足時のエラーメッセージを適切にフロントへ返し、ユーザーに通知すること。

# 開発ステップ（指示）
1. フォルダ構成の作成と Docker / DevContainer 設定ファイルの生成。
2. バックエンドのモデル定義、バリデーション、API実装。
3. フロントエンドのUIコンポーネント（ホバーアクション含む）の実装。

まずは、プロジェクトのディレクトリ構成案と `docker-compose.yml` から作成を開始してください。
